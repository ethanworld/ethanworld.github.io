    分布式存储¶

    约 1754 个字 1 张图片 预计阅读时间 6 分钟

    分布式存储框架主要分为==分布式文件系统、对象存储和块存储三类==。主流的开源框架包括 Ceph（全能）、HDFS（大数据首选）、MinIO（对象存储）、GlusterFS（分布式文件系统）等，适用于海量数据存储、云原生应用及大数据分析。

    具体分类和代表系统如下：

    1、分布式文件系统 (Distributed File Systems)

    适用于海量文件存储、共享。

    • CephFS: Ceph组件之一，支持POSIX兼容。
    • HDFS (Hadoop Distributed File System): 大数据生态核心，擅长超大文件存储和流式访问。
    • GlusterFS: 易于部署，适合文档存储、备份。
    • MooseFS (MFS): 轻量级、高可用，支持FUSE挂载。
    • JuiceFS: 兼容S3的对象存储文件系统。 

    2、对象存储 (Object Storage)

    适用于非结构化数据、云存储、备份。

    • MinIO: 高性能对象存储，兼容Amazon S3接口，易于容器化部署。
    • Swift: OpenStack的开源对象存储。
    • Ceph RGW (Rados Gateway): 提供S3/Swift兼容的对象存储。 

    3、分布式块存储 (Block Storage)

    适用于虚拟机、数据库、云硬盘。

    • Ceph RBD (Rados Block Device): 提供高性能、可动态扩展的块设备。
    • Longhorn: 云原生环境（K8s）下的分布式块存储。
    • FusionStorage: 华为推出的高性能国产块存储系统。

    4、分布式KV/数据库存储

    适用于结构化数据、高并发读写。

    • TiKV: 分布式事务型键值数据库。
    • Apache Cassandra: 最终一致性的NoSQL数据库。

    选型参考

    • 全面、通用： Ceph。
    • 大数据、分析： HDFS。
    • 云原生、对象存储： MinIO。
    • 图像、小文件： FastDFS。

    分布式存储¶

    分布式系统架构设计原理与实战：分布式存储系统- 掘金

    核心特点：将数据切分成多个块，分布在多个存储节点上。实现高可靠（数据冗余、故障容错、快恢）、高可扩展。

    核心难点：数据一致性模型，通过复制、副本、数据同步机制等保证多节点的数据一致性。

    核心特性：

    • 数据切分：将要存储的数据切分成较小的块或对象，每个块或对象都有唯一的标识符。
    • 冗余备份：将切分的数据块或对象复制到多个节点上，以提高数据的可靠性和容错性。冗余副本可以分布在不同的节点上，以防止单点故障。
    • 容错和恢复：分布式存储系统通常具有容错机制，可以检测和纠正数据损坏或节点故障。当节点故障时，系统可以自动将数据从冗余副本中恢复。
    • 数据分布和负载均衡：将数据块或对象分布到不同的节点上，以实现负载均衡和提高性能。通常使用一致性哈希等算法来确定数据在节点之间的分布。
    • 元数据管理：维护数据的元数据，包括数据的位置、副本数、访问权限等信息。元数据可以存储在专门的元数据服务器上，也可以分布在各个存储节点上。
    • 数据访问和一致性：通过标识符或其他方式来访问存储的数据。分布式存储系统通常提供读写接口和一致性保证机制，以确保数据的一致性和可靠性。
    • 扩展性和动态扩展：分布式存储系统可以根据需求动态扩展，添加新的存储节点来增加存储容量和性能。

    核心算法：

    • 数据分布和定位：决定数据存放在哪个节点，要求尽可能平均分布，且在扩容时减少数据迁移。

      • 一致性哈希：将哈希空间抽象为一个圆环，节点和数据通过哈希函数映射到环上。顺时针查找，数据存放在最近的节点。当节点增减时，仅影响局部数据，避免全量数据重新映射。

      • CRUSH： 专用于Ceph分布式存储，通过计算而非查找表来确定数据位置。它利用物理结构（机架、主机）进行规则映射，实现了去中心化的高性能寻址。

    • 分布式一致性算法：在多副本存储场景下，保证所有副本的数据状态在同一时间点对用户表现一致。

      • Paxos 算法： 经典的强一致性算法。通过提案（Proposal）和少数服从多数的投票机制，确保在节点故障或网络分区时达成一致。

      • Raft 算法： 更易理解与实现的强一致性算法。它将一致性分解为领导者选举、日志复制和安全性三个部分，目前被主流系统（如Etcd、TiKV）广泛使用

      • ZAB (ZooKeeper Atomic Broadcast)： 专为ZooKeeper设计。借鉴了Paxos，核心是保证各节点之间状态转换的顺序性和原子性，适合在强一致性需求的元数据管理场景

    • 数据高可用和冗余算法：确保单个或多个节点损坏时数据不丢失。

      • 副本复制 (Replication)： 简单直接，将数据复制多份存放在不同节点（如HDFS常用的3副本）。

      • 纠删码 (Erasure Coding, EC)： 相比副本策略，EC能以更低的技术冗余度实现高可靠性。它通过数据切片和校验和计算（如RS编码）来重建丢失的数据。

    • 数据清理和平滑算法：

      • 数据重平衡 (Rebalancing)： 当系统扩容或负载不均时，自动移动数据以达到节点负载均衡。

      • 碎片回收 (Garbage Collection)： 清理分布式系统因更新、删除操作产生的垃圾数据，释放存储空间。

    一致性哈希¶

    图解一致性哈希算法，看这一篇就够了！ -阿里云开发者社区

    作用：通过将节点（虚拟节点）+ 数据均映射到hash环，数据按顺时针找最近的节点，解决数据分布和定位问题，该算法能尽量减少因节点缩容或者扩容带来的数据搬迁影响。

    EC纠偏码原理¶

    EC 纠删码用线性代数生成 K 个校验块，能容忍任意 K 块失效，用远低于多副本的空间实现高可靠，是现代分布式存储的核心冗余技术。

    EC(4,2)、EC(6,3)

    选择两种数据保护方式，就需要根据业务情况因地制宜。

    多副本技术**可以更好的保证数据的安全性，提升读写效率。**一般适用于关键业务，对性能要求比较高，网络对数据出错容忍度极低的业务，如数据库等。

    纠删码则可以提高存储系统利用率，降低成本。一般适用于其他非关键性业务，如视频、影像存储等海量存储场景，部分数据块的损坏不会影响信息的传递。
